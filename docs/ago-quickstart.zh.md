# Ago 快速开始

从零到看着一个目标自己跑完，大约五分钟。

---

## 1. 它到底做什么

你说一句话，Ago 把它变成一张持久的任务图，然后自己跑完：拆解、排期、隔离执行、独立验收、把通过的改动合到一个自己的 git 分支上。

关键在于「说完之后不用再说第二句」。Ago 只在这些情况下打断你：

- 要推送、发布、部署、花钱；
- 破坏性操作；
- 需要凭据；
- 不可逆的产品方向决定；
- 目标本身有仓库证据也解决不了的歧义；
- 超出你设定的预算、时间或权限；
- 自动修复和有界重试都用尽了。

其它一切——读报告、判断可不可信、写修复要求、说「继续」——都是机械劳动，由 Ago 自己做。

## 2. 需要什么

| 依赖 | 为什么 |
|------|--------|
| `git` | 每次尝试都在独立的 git worktree 里跑 |
| `go` | 示例仓库是一个真的 Go 项目，Ago 要真的跑它的测试 |
| 一个 OpenAI 兼容端点（可选） | 只有 `--executor relay` 需要 |

不需要账号，不需要配置文件，不需要联网（离线模式）。

## 3. 三十秒跑起来

```bash
go build -o ago ./cmd/ago
./ago demo --executor fake
```

输出：

```
示例仓库已创建：~/.ago/demo/greeter
Ago is ready
UI:       http://127.0.0.1:61604
Database: ~/.ago/demo/ago.db
Mode:     fake
Scenario: success
Note:     fake mode runs a fixed plan with scripted outcomes; no model decides anything.

目标：为这个示例 CLI 增加 version 子命令，补充 README 使用说明，增加自动化测试，并生成完成报告。
打开看板：http://127.0.0.1:61604
按 Ctrl+C 结束。
```

打开那个地址，看板会实时推进。加 `--open` 可以自动开浏览器。

**注意那行 Note。** 离线模式跑的是固定计划和脚本化结果——它验证的是机器本身（任务图、租约、独立验收、集成分支），不是模型的判断力。想看模型真的做决定，用下一节。

## 4. 用真实模型跑

```bash
export AGO_RELAY_BASE_URL=http://127.0.0.1:8317/v1
export AGO_RELAY_API_KEY=...          # 只能从环境变量给，见第 8 节
./ago demo --executor relay
```

启动前会先做一次真实的小调用检查端点和凭据能不能用。不能用就在这里报错，而不是让你等到第四分钟看到一个关于租约的失败任务。

可选地指定每个角色的模型：

```bash
export AGO_PLANNER_MODEL=claude-sonnet-5
export AGO_EXECUTOR_MODEL=claude-sonnet-5
export AGO_VERIFIER_MODEL=claude-sonnet-5
```

## 5. 换一个目标

```bash
./ago demo --executor relay --goal "把 greet 子命令改成支持多个名字，并补充测试"
```

计划是模型现场生成的，不是模板。同一个目标跑两次，任务数和依赖结构会不一样。

## 6. 看板上有什么

- **列**：ready / running / verifying / integrating / retry-wait / passed / failed
- **任务抽屉**：验收标准、每次尝试的记录、命令和它们真实的退出码、测试结果、变更文件的前后哈希、可下载的工件
- **注意队列**：需要你决定的事情。空的就是空的——那才是正常状态
- **暂停/继续**：暂停是持久的，重启之后还是暂停

## 7. 中断和恢复

Ctrl+C 是安全的：当前这一轮调度会跑完，已经领取的尝试不会被半路丢掉。

再跑一次同样的命令，它会接着上次的地方走：

```
沿用已有的示例仓库：~/.ago/demo/greeter
看板已存在，从上次停下的地方继续。
```

想从头开始：`--reset`。它不是 `rm -rf`：

- 只清理这几个固定名字（`ago.db*`、`greeter`、`artifacts`、`worktrees`、`integration`），同一个目录里的其它东西都会留下 —— 包括成功清理之后；
- 但光靠这个还不够，因为 `artifacts` 和 `integration` 是很普通的目录名。真正的关口是**认领**：Ago 只会把「不存在」「空的」「里面只有 Ago 自己那几项」的目录当作演示目录。`--state` 指到一个装着你自己东西的目录，Ago 会直接拒绝运行，也不会写下归属标记 —— 没有标记，`--reset` 就永远动不了它；
- 归属标记是 `.ago-demo-state`，内容也要对得上；只是重名的空文件、或者指向别处的符号链接，都会被拒绝；
- `--state` 本身是符号链接会被拒绝（你说的是一个目录，删的会是另一个）；路径中间的符号链接会被解析成真实路径，所有删除都发生在那个真实目录里，链接是被摘掉而不是被跟进去的；
- 根目录、主目录、包含主目录的目录、太靠近根的路径，以及 git 仓库（含 worktree 和 submodule，它们的 `.git` 是文件而不是目录），一律拒绝；
- 判断在任何 preflight 之前就做完，删除在所有 preflight 之后才执行 —— 凭据错了不会先毁掉你已有的状态再报错。

## 8. 凭据

`AGO_RELAY_API_KEY` **只能**通过环境变量传，命令行不接受它。原因很直接：命令行参数会留在 shell 历史里，也会出现在进程列表里，机器上任何一个进程都能看到。

凭据不会进入任何提示词、任务、事件、日志或数据库。这一点是有测试守着的：端到端测试会在 `ago.db` 和 `ago.db-wal` 里搜索密钥的字面值。

## 9. 它不会碰你的东西

- 每次尝试在独立的 detached worktree 里跑（不是 `-b`，那会往你的仓库里写真的分支，而 `worktree remove` 不会删掉它们）
- 通过验收的改动只会写到 `refs/heads/ago/*`，绝不会碰你的分支
- 从不 push
- 验证命令（`go test`、`make`）只用来验证：它们产生的文件会被丢弃并记录，不会成为交付内容
- 检查跑在工作树之外的 HOME 里，所以工具缓存不会污染仓库

## 10. 常见问题

**「端口已被占用」** —— 用 `--listen 127.0.0.1:0` 自动挑一个空闲端口，或换一个端口。

**「找不到 git / go」** —— 装上。示例仓库要真的编译和跑测试，假装不了。

**「无法通过 ... 调用模型」** —— 检查 `AGO_RELAY_BASE_URL`、`AGO_RELAY_API_KEY` 和模型名。注意有些本地代理不会转发 system 消息；Ago 会把指令同时放进 user 消息，所以这类代理也能用。

**任务卡在 retry-wait** —— 那是有界退避，不是卡住。Ago 会等到时间到再重试，退避次数用完会作为一条决定进入注意队列。

**注意队列里出现了一条决定** —— 那就是设计意图：自动修复和有界重试都用尽了，或者遇到了机器不该独自决定的事。抽屉里写着发生了什么和建议怎么做。

**`ago-server` 还在吗** —— 在，作为兼容和开发入口：`ago-server serve` 只起服务不种目标，`ago-server demo` 等价于 `ago demo`。两者跑的是同一份 `internal/agoserve`，没有第二套编排。用户路径是 `ago demo`。
